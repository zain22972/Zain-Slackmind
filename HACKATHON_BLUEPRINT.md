# Hackathon Product Blueprint: Intelligent Slack Knowledge Base

This comprehensive technical blueprint outlines the exact specifications, requirements, architecture, and step-by-step 3-hour execution plan for building the **Intelligent Slack Knowledge Base**. This document is designed to serve as both your architecture reference and your rapid engineering checklist during the hackathon.

---

## 1. Project Requirements & Product Specification (`REQUIREMENTS.md`)

### 1.1 Executive Product Goal
The **Intelligent Slack Knowledge Base (ISKB)** bridges the critical organizational gap where institutional knowledge lives in disconnected silos (PDFs, URLs, local drives) while teams live in Slack. By creating an autonomous conversational intelligence layer directly within the primary workspace—coupled with a lightweight administrative control dashboard—employees can instantly retrieve accurate, fully-grounded, and context-aware insights without interrupting workflows or reinventing the wheel.

### 1.2 Core Functional Requirements

#### A. Multi-Format Ingestion Engine
* **Source Formats:** Native support for parsing textual content from `.pdf` documents, raw text (`.txt`) files, and web page URLs.
* **Chunking & Processing:** Automated document parsing, semantic text chunking (fixed size with overlap to maintain relational context), and generation of vector representations.
* **Metadata Tagging:** Automatic injection of document attributes during ingestion, including filename, unique source URL/path, timestamp, and security classification level (`Org`, `Team`, or `Private`).

#### B. Slack Interaction Interface
* **Natural Language Workspace Q&A:** Users trigger the intelligence engine via standard `@mention` queries within any channel or direct message thread where the bot is present (e.g., `@KnowledgeBot What is our paternity leave policy?`).
* **Verifiable Source Citations:** Every response generated by the system must include distinct footnotes referencing the exact source documents extracted from the knowledge base to prevent blind trust and allow rapid document auditing.
* **On-Demand Summarization:** Dedicated Slack Slash Command (`/summarize [document_name_or_url]`) providing an immediate, high-density, 3-to-5 bullet-point TL;DR summarizing the document context without modifying the database state.
* **Strict Hallucination Mitigation:** The model must operate strictly on a deterministic closed-domain paradigm. If the vector retrieval stage does not yield highly relevant semantic matches corresponding to the prompt, the engine must safely output a standard gracefully-handled refusal: *"I cannot find this information in the knowledge base."* It must never default to generalized public pre-training knowledge.

#### C. Multi-Scope Access Security Controls
* **Scoped Knowledge Access Layers:** Isolation of knowledge domains using structural database metadata tagging:
    * `Org`: Universally searchable by any authenticated team member in the workspace.
    * `Team`: Restructured access limited strictly to queries originating from specific designated Slack channels.
    * `Private`: User-bound knowledge scopes isolated completely to queries originating from that matching unique Slack `user_id`.
* **Dynamic Context Filtering:** Runtime query modification ensuring that the retrieval step filters vectors explicitly based on origin context (`user_id` and `channel_id`).

#### D. Real-Time Admin Dashboard (React Web App)
* **Visual Index Library:** A unified web view displaying a high-density grid/table listing all current ingested files, complete with metadata (file size, timestamp, scope) and its underlying operational state (e.g., `Processing`, `Indexed`, or `Failed`).
* **Direct Upload Portal:** A modern web drag-and-drop file interface allowing rapid, bulk-upload execution of files directly into the active processing pipeline.

## 2. Technical Stack & Architectural Design

To maximize execution velocity within a strict **3-hour window**, the architecture focuses on highly optimized, out-of-the-box developer ecosystems. It balances Python's dominant machine-learning capabilities with JavaScript's immediate interface rendering performance.

```

                                │ (Webhook Events)           │ (Slack HTTP POST)
                                ▼                            │
 ┌───────────────────────────────────────────────────────────┴───────────────┐
 │                       FASTAPI BACKEND SYSTEM                              │
 │                                                                           │

 │              │                                    │                       │
 │              ▼                                    │                       │
 │    LangChain RAG Processing                       │ (JSON Payload / multipart)
 │   (OpenAI Embeddings / Chat)                      │                       │
 └──────┬───────────────▲────────────────────────────┼───────────────────────┘
        │               │                            │
```

### 2.1 Technology Matrix
* **Frontend Interface:** **React (via Vite)** for zero-configuration builds, styled using **Tailwind CSS** and clean layouts for rapid component construction without long asset compilation cycles.
* **Backend Orchestrator:** **FastAPI (Python 3.10+)** for ultra-fast performance, automatic interactive API documentation, and async webhook response execution natively optimized for Slack’s tight 3000ms response timeout.
* **Orchestration Framework:** **LangChain** or **LlamaIndex** to abstract complex text ingestion, chunking math, and context injection loops into high-level declarative components.
* **Large Language Model & Embeddings:** **OpenAI API (`gpt-4o-mini`)** ensuring ultra-low transactional latency, exceptional reasoning capabilities on strict prompt directives, and low cost; paired with **`text-embedding-3-small`** for fast, high-density 1536-dimensional vector maps.
* **Vector Engine:** **Pinecone (Serverless Free Tier)** or a local instance of **ChromaDB / DuckDB** configured for metadata-filtered index searches.

## 3. High-Velocity Implementation Roadmap (3-Hour Timeline)

### Phase 1: Environment Provisioning, Slack Configuration & Scopes (00:00 - 00:30)
1. **Repository Setup:** Run Vite and FastAPI boilerplates instantly. Merge local structures so that the frontend and backend are housed within a unified single-repository codebase.
2. **Slack Portal Setup:** Navigate to [api.slack.com/apps](https://api.slack.com/apps), select "Create New App from Scratch", and bind it to your development sandbox workspace.
3. **Configure Access Scopes:** Navigate to **OAuth & Permissions** and provision the following specific OAuth scopes:
    * `app_mentions:read` (Allows the app to listen to explicit query mentions)
    * `chat:write` (Allows the app to post formatted answers back to channels)
    * `commands` (Enables provisioning of custom slash triggers)
    * `files:read` (Allows handling of document payloads sent within Slack workspace)
4. **Proxy Local Dev Server:** Initialize an `ngrok` tunnel mapping your local FastAPI port (`ngrok http 8000`). Copy the HTTPS forwarding string, navigate to **Event Subscriptions** in Slack, toggle the feature "On", and input `https://[your-ngrok-subdomain].ngrok-free.app/slack/events` into the Request URL. Confirm verification status matches "Verified".

### Phase 2: Ingestion Pipeline & Web Embeddings (00:30 - 01:15)
1. **Write Ingestion Core:** Create a document chunker using LangChain's `RecursiveCharacterTextSplitter`. Set constraints strictly to `chunk_size=1000` tokens with a `chunk_overlap=200` token boundary.
2. **Vector DB Binding:** Write an ingestion function that maps text chunks directly into a Pinecone vector index pre-configured with 1536 dimensions utilizing Cosine distance metrics.
3. **Database Metadata Strategy:** Ensure every uploaded document structure automatically injects clear query filters.
    ```python
    metadata = {
        "source": doc.name,
        "scope": "org", # Options: "org" | "team_[id]" | "user_[id]"
        "timestamp": time.time()
    }
    ```

### Phase 3: Slack Webhook Listening & Core RAG Integration (01:15 - 02:15)
1. **Initialize Slack Bolt Wrapper:** Instantiate the `slack_bolt.App` pipeline utilizing your `SLACK_BOT_TOKEN` and `SLACK_SIGNING_SECRET`.
2. **Construct RAG Query Pipeline:** Combine LangChain's vector retriever with a zero-temperature `gpt-4o-mini` model instance.
3. **Enforce Deterministic Strict Prompt Guardrails:**
    ```text
    SYSTEM PROMPT:
    You are an intelligent organizational assistant. Your objective is to solve the user's inquiry utilizing ONLY the factual data explicitly provided in the context blocks below.
    
    Context:
    {context}
    
    Operational Guardrails:
    1. If the provided context is completely insufficient or completely irrelevant to resolve the inquiry, state exactly: "I cannot find this information in the knowledge base." Do not attempt to formulate generalized answers.
    2. For every answer provided, explicitly append the names of the source documents utilized under a clear "Sources Used:" footer section.
    ```
4. **Deploy Mention Listener:** Write the async event hook for `@app.event("app_mention")` to extract user text, stripping out the raw bot mentions, pass the text to your RAG engine, and stream or reply the output string back to the channel thread using `say()`.

### Phase 4: Security Access Scoping & Slash Commands (02:15 - 02:45)
1. **Implement Runtime Filtering:** Extract the payload variables `event['user']` and `event['channel']` from incoming Slack webhooks. Inject a structural dynamic filter array parameter directly into your Pinecone/Chroma search retriever:
    ```python
    search_kwargs={
        "k": 3,
        "filter": {"scope": {"$in": ["org", f"team_{event['channel']}", f"user_{event['user']}"]}}
    }
    ```
2. **Expose `/summarize` Endpoint:** Implement the `/summarize` Slack Command listener. Fetch target document segments using simple text-matching rules or direct vector fetches, group text, and send to an LLM chain optimized for high-density generation.

### Phase 5: Pipeline Seeding, Validation & Presentation Prep (02:45 - 03:00)
1. **Data Priming:** Execute a pre-written batch script to fully seed 48 test documents (policies, engineering manuals, fake corporate reports) into the vector engine. Verify index counts read exactly $\geq 48$.
2. **Live Test Validation:** Retain 2 specific target documents separate from the seed files. Use them to execute a live demonstration of your pipeline's ingestion speed and show how new data becomes searchable in real-time.
3. **Check Error Handlers:** Run an explicit test query on an unrelated topic (e.g., "What is the capital of France?") to ensure your bot outputs the designated refusal string instead of hallucinating.

## 4. Key Hackathon Optimization Shortcuts
* **Avoid Complex Slack File Sync:** Downloading media directly through Slack's secure binary gateways requires heavy streaming multi-part architecture. For the hackathon demo, prioritize loading files through your **React Dashboard**, and let the Slack interface focus on immediate Q&A and text summary queries.
* **Leverage Lightweight Ephemeral Memory:** Skip configuring external persistent Redis databases for conversation history. Keep it simple: use single-turn QA query structures or pass the last 2-3 historical items directly inside the request payload text wrapper.
* **Mock Complex Auto-Tagging:** Instead of training or layering asynchronous classification models to categorize document groupings, have your ingestion LLM process the document summary text for 1 second during upload to output 3 distinct tag words, appending them straight to the vector's metadata.
